A Clean Architecture é uma filosofia de design de software que enfatiza a separação de preocupações e a manutenibilidade. Ela estrutura uma aplicação em camadas, cada uma com responsabilidades distintas, para criar sistemas independentes de frameworks, bancos de dados ou tecnologias de UI.

### Princípios e Camadas Fundamentais

*   **Separação de Preocupações:** Cada camada possui uma responsabilidade distinta, tornando a aplicação mais fácil de gerenciar, entender e testar.
*   **Regra da Dependência:** As dependências entre as camadas devem sempre apontar para dentro. As camadas internas não devem ter conhecimento das camadas externas. Isso garante que as mudanças nas camadas externas não afetem as camadas internas, aumentando a modularidade e a manutenibilidade.
*   **Camadas:**
    *   **Entities (Camada de Domínio):** A camada mais interna, contendo as regras de negócio e entidades centrais. É agnóstica à tecnologia e define os conceitos fundamentais da aplicação.
    *   **Use Cases (Camada de Aplicação):** Esta camada contém as regras de negócio específicas da aplicação e orquestra o fluxo de dados entre entidades e sistemas externos. Ela define o que a aplicação faz.
    *   **Interface Adapters:** Esta camada é responsável por converter dados de sistemas externos (como UI ou bancos de dados) para um formato utilizável pelas camadas internas, e vice-versa.
    *   **Frameworks & Drivers (Camada de Apresentação/Infraestrutura):** A camada mais externa, lidando com preocupações externas como UI frameworks, bancos de dados e serviços de terceiros.

### Configuração e Bootstrap na Clean Architecture

O processo de bootstrap de uma aplicação construída com Clean Architecture envolve a configuração inicial e as dependências para que a aplicação comece a funcionar. Isso frequentemente inclui:

*   **Injeção de Dependência (DI):** Um aspecto crucial da Clean Architecture, a DI é usada para desacoplar componentes e gerenciar dependências. Ela permite testes mais fáceis e flexibilidade ao injetar implementações de interfaces em construtores.
    *   A camada de Aplicação frequentemente contém uma classe de Injeção de Dependência para separar ainda mais as preocupações.
    *   As camadas de Apresentação e Infraestrutura tipicamente registram suas dependências, frequentemente referenciando a camada de Aplicação para elementos comuns.
*   **Estrutura do Projeto:** Um layout de projeto consistente é vital. Estruturas comuns incluem a separação de projetos/pastas para Core (Domínio e Aplicação), Infraestrutura e Apresentação.
    *   Algumas abordagens mesclam as camadas Core e Aplicação em um único projeto "Core" para simplificar.
*   **Gerenciamento de Configuração:** A "camada de Bootstrap" ou arquivos de configuração lidam com a configuração inicial da aplicação, incluindo a configuração de DI, localização e outras configurações globais.
*   **Ponto de Entrada:** O ponto de entrada da aplicação (por exemplo, `Program.cs` em .NET, `main.tsx` em React) é responsável por chamar o processo de bootstrap e configurar o pipeline de requisições HTTP ou outros serviços da aplicação.
*   **Testes:** A Clean Architecture promove o teste independente de cada camada, o que é um benefício significativo. Mocking e isolamento são práticas-chave para testes.

### Exemplos em Diferentes Tecnologias

*   **.NET Core:** Muitos recursos fornecem orientação sobre como configurar a Clean Architecture em .NET Core, frequentemente usando templates e demonstrando como configurar cada camada (Domínio, Aplicação, Infraestrutura, API).
*   **PHP:** A Clean Architecture pode ser implementada sem depender fortemente de frameworks MVC, usando ferramentas como Composer para gerenciamento de dependências e contêineres de DI.
*   **React/Next.js:** Os princípios da Clean Architecture podem ser aplicados a aplicações frontend, com uma camada de "bootstrap" lidando com configurações e configuração inicial.
*   **Go:** A Clean Architecture em projetos Go envolve a estruturação do código em entidades centrais, casos de uso, adaptadores de interface e frameworks/drivers, com responsabilidades claras para cada camada.

Embora a Clean Architecture ofereça benefícios significativos como melhor legibilidade, testabilidade e escalabilidade, ela pode introduzir uma sobrecarga e complexidade iniciais, especialmente para aplicações muito pequenas.
