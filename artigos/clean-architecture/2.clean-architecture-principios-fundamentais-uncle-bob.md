A Clean Architecture é uma filosofia de design de software que enfatiza a separação de preocupações para criar sistemas independentes, testáveis e de fácil manutenção, que sejam resilientes a mudanças tecnológicas e requisitos de negócio.

### Princípios Fundamentais da Clean Architecture

O cerne da Clean Architecture gira em torno da **Regra da Dependência**, que afirma que as dependências do código-fonte só podem apontar para dentro. Isso significa que as camadas internas não devem ter conhecimento das camadas externas, garantindo que as mudanças nos componentes externos não afetem a lógica de negócio central. Essa independência é alcançada através do uso de interfaces, onde as camadas externas definem interfaces das quais as camadas internas dependem.

As principais características e objetivos da Clean Architecture incluem:

*   **Testabilidade:** Ao isolar a lógica de negócio, a funcionalidade central pode ser testada completamente sem dependências externas.
*   **Independência:** Visa a independência de frameworks, bancos de dados e tecnologias de UI.
*   **Flexibilidade e Manutenibilidade:** A arquitetura permite que diferentes partes de uma aplicação evoluam independentemente, tornando-a adaptável a requisitos em constante mudança.
*   **Separação de Política e Detalhes:** As regras de negócio (políticas de alto nível) devem ser separadas dos detalhes técnicos (implementações de baixo nível).

A Clean Architecture tipicamente consiste em camadas concêntricas, cada uma com responsabilidades específicas:

*   **Entities (Camada mais interna):** Representa a lógica de negócio pura e os objetos de domínio, independentes de qualquer fonte de dados ou framework de UI.
*   **Use Cases (Interactors):** Contém as regras de negócio específicas da aplicação e orquestra o fluxo de dados para e das entidades.
*   **Interface Adapters:** Converte dados entre o formato exigido pelos casos de uso e entidades, e o formato usado por agências externas, como bancos de dados ou a UI.
*   **Frameworks and Drivers (Camada mais externa):** Inclui frameworks de terceiros, ferramentas e mecanismos de entrega.

### Princípios SOLID na Clean Architecture

Os princípios SOLID são um conjunto de diretrizes para o design orientado a objetos que são fundamentais para alcançar a Clean Architecture. Eles ajudam a criar código modular, flexível e de fácil manutenção. Esses princípios foram popularizados na comunidade de desenvolvimento de software e são aplicados no nível intermediário (nível de classe/módulo) da estrutura do software.

Os princípios SOLID são:

*   **S - Single Responsibility Principle (SRP - Princípio da Responsabilidade Única):** Um módulo (ou classe) deve ter um, e apenas um, motivo para mudar. Em um nível arquitetural, isso pode ser definido como "um módulo deve ser responsável por um, e apenas um, ator."
*   **O - Open/Closed Principle (OCP - Princípio Aberto/Fechado):** Entidades de software (classes, módulos, funções, etc.) devem ser abertas para extensão, mas fechadas para modificação. Isso significa que novas funcionalidades podem ser adicionadas sem alterar o código existente e funcionando.
*   **L - Liskov Substitution Principle (LSP - Princípio da Substituição de Liskov):** Objetos em um programa devem ser substituíveis por instâncias de seus subtipos sem alterar a correção desse programa. Isso promove o polimorfismo e garante que as classes derivadas possam substituir suas classes pai.
*   **I - Interface Segregation Principle (ISP - Princípio da Segregação de Interfaces):** Clientes não devem ser forçados a depender de interfaces que não usam. Muitas interfaces específicas do cliente são melhores do que uma interface de propósito geral.
*   **D - Dependency Inversion Principle (DIP - Princípio da Inversão de Dependência):** Módulos de alto nível não devem depender de módulos de baixo nível. Ambos devem depender de abstrações. Abstrações não devem depender de detalhes. Detalhes devem depender de abstrações. Este princípio é considerado um pilar da Clean Architecture, permitindo a "arquitetura de plug-in" ao colocar limites entre componentes de alto e baixo nível.

Ao aderir a esses princípios, a Clean Architecture permite a construção de sistemas que são não apenas funcionais, mas também duradouros e adaptáveis às mudanças, garantindo que o investimento em software continue a gerar valor ao longo do tempo.
