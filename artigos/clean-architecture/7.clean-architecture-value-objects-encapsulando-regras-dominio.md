Value Objects são blocos de construção fundamentais no design de software, especialmente no contexto da Clean Architecture e do Domain-Driven Design (DDD). Eles representam aspectos descritivos do domínio que não possuem identidade conceitual e são definidos exclusivamente por seus atributos.

### O que são Value Objects?

Value Objects são objetos imutáveis que representam um único valor ou conceito dentro do domínio. Ao contrário das entidades, eles não possuem um identificador único; sua igualdade é determinada pela igualdade de seus valores encapsulados. Por exemplo, um objeto `Dinheiro` que consiste em um `valor` e uma `moeda` é um Value Object. Dois objetos `Dinheiro` são considerados iguais se seus valores e moedas forem os mesmos, independentemente de serem a mesma instância na memória.

### Imutabilidade em Value Objects

Uma característica central dos Value Objects é sua imutabilidade. Uma vez que um Value Object é criado, seu estado interno não pode ser alterado. Se uma modificação for necessária, uma nova instância do Value Object é criada com os valores atualizados. Essa imutabilidade oferece vários benefícios:

*   **Previsibilidade e Consistência:** Garante que o estado do objeto permaneça constante após a criação, tornando o código mais fácil de raciocinar e reduzindo a probabilidade de efeitos colaterais inesperados.
*   **Segurança de Thread:** Objetos imutáveis são inerentemente seguros para threads, pois seu estado não pode ser modificado por várias threads simultaneamente.
*   **Comparação Mais Fácil:** A igualdade pode ser determinada comparando os valores de seus atributos, simplificando a lógica de comparação.

### Regras de Domínio e Validação em Value Objects

Value Objects são excelentes para encapsular regras de domínio e invariantes relacionadas ao seu conceito específico. Isso significa que um Value Object pode impor sua própria validade no momento da criação, impedindo a instanciação de objetos inválidos. Por exemplo, um Value Object `PrimeiroNome` poderia impor uma restrição de comprimento máximo ou garantir que não esteja vazio. Essa autovalidação garante que o modelo de domínio esteja sempre em um estado válido, reduzindo erros e melhorando a confiabilidade do código.

### Value Objects na Clean Architecture

Na Clean Architecture, os Value Objects residem na camada mais interna, a **Camada de Domínio**. Esta camada contém a lógica de negócio central e os tipos, e não deve depender de nada fora de si mesma. Ao colocar os Value Objects aqui, eles permanecem independentes de preocupações externas, como interfaces de usuário, bancos de dados ou frameworks específicos.

A Clean Architecture enfatiza a separação de preocupações. Value Objects, juntamente com Entidades e Domain Events, formam o coração da camada de domínio, representando os conceitos fundamentais do negócio. Eles contribuem para uma arquitetura flexível, mantenível e testável, garantindo que as regras de negócio sejam encapsuladas e isoladas.

A perspectiva da Clean Architecture é que a camada mais interna, embora amplamente rotulada como "Entities", abrange todo o modelo de domínio, que inclui Value Objects, Entidades, Agregados e Domain Services de uma perspectiva DDD. O ponto chave dos princípios da Clean Architecture é que as regras de negócio e as estruturas de dados centrais (incluindo Value Objects) devem estar no centro da aplicação, livres de dependências externas. Essa adesão à Regra da Dependência (camadas internas não devem depender de camadas externas) garante que as mudanças em preocupações externas não impactem o domínio central.

Em essência, Value Objects na Clean Architecture são objetos imutáveis, autovalidáveis e sem identidade que encapsulam conceitos de domínio específicos, residindo na camada de domínio central para impor regras de negócio e manter uma separação limpa de preocupações.
