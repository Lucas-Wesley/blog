Clean Architecture e Hexagonal Architecture (também conhecida como Ports and Adapters) são padrões arquiteturais de software que visam criar sistemas manteníveis, testáveis e flexíveis, separando a lógica de negócio central de preocupações externas. Embora compartilhem objetivos semelhantes, eles abordam a separação de preocupações com ênfases ligeiramente diferentes.

### Arquitetura Hexagonal (Ports and Adapters)

A Arquitetura Hexagonal, introduzida por Alistair Cockburn, foca em isolar a lógica de negócio central (o "core" ou "domínio") de dependências externas como bancos de dados, interfaces de usuário e serviços de terceiros. O nome "hexagonal" vem da representação visual do componente da aplicação como um hexágono, com vários sistemas externos interagindo com ele através de seus lados.

**Conceitos Chave:**

*   **Core (Lógica de Domínio):** Este é o coração da aplicação, contendo as regras de negócio puras, entidades e casos de uso. É agnóstico à tecnologia e independente de sistemas externos.
*   **Ports (Portas):** São interfaces abstratas ou contratos que definem como o core interage com o mundo exterior. Eles especificam as operações que o core pode realizar ou que sistemas externos podem solicitar do core. Existem dois tipos de portas:
    *   **Driving Ports (Inbound):** Usadas por atores externos (por exemplo, UI, API) para "dirigir" o core da aplicação.
    *   **Driven Ports (Outbound):** Usadas pelo core da aplicação para interagir com sistemas externos (por exemplo, bancos de dados, APIs externas).
*   **Adapters (Adaptadores):** São implementações concretas das portas. Eles atuam como uma ponte entre os sistemas externos e a lógica central, traduzindo detalhes tecnológicos específicos externos para um formato que o core entende, e vice-versa.
    *   **Driving Adapters (Primários/Inbound):** Implementam portas de entrada e lidam com a entrada de fontes externas (por exemplo, um controlador web, uma CLI).
    *   **Driven Adapters (Secundários/Outbound):** Implementam portas de saída e lidam com a saída para sistemas externos (por exemplo, um repositório de banco de dados, um cliente de serviço externo).

**Benefícios da Arquitetura Hexagonal:**

*   **Separação de Preocupações:** Isola claramente a lógica de negócio dos detalhes de infraestrutura.
*   **Testabilidade:** A lógica de negócio central pode ser testada independentemente usando implementações mock de portas, pois não possui dependências diretas de sistemas externos.
*   **Flexibilidade e Adaptabilidade à Mudança:** Sistemas externos (por exemplo, troca de bancos de dados, mudança de frameworks de UI) podem ser substituídos com impacto mínimo na lógica central, desde que os adaptadores adiram às interfaces da porta.
*   **Modularidade:** Incentiva o desenvolvimento de componentes fracamente acoplados que podem ser desenvolvidos e mantidos independentemente.

### Clean Architecture

Clean Architecture é um conceito mais amplo que integra princípios de vários padrões arquiteturais, incluindo Arquitetura Hexagonal, Onion Architecture e Arquitetura em Camadas. Ela enfatiza uma estrutura em camadas com uma rigorosa "Regra da Dependência", o que significa que as dependências só podem apontar para dentro, das camadas externas para as camadas internas.

**Componentes Chave (Camadas do mais interno para o mais externo):**

1.  **Entities:** Encapsulam as regras de negócio de toda a empresa e são a parte mais estável da aplicação.
2.  **Use Cases (Interactors):** Contêm as regras de negócio específicas da aplicação e orquestram o fluxo de dados para e das entidades.
3.  **Interface Adapters:** Adaptam dados do mundo externo para o formato exigido pelos casos de uso e entidades, e vice-versa. Esta camada inclui controladores, gateways (como gateways de banco de dados) e apresentadores. É aqui que o conceito de "Ports and Adapters" se encaixa amplamente na Clean Architecture.
4.  **Frameworks and Drivers:** A camada mais externa, consistindo em detalhes como frameworks web, bancos de dados, frameworks de UI e outras ferramentas externas.

**Relação entre Clean Architecture e Arquitetura Hexagonal:**

A Arquitetura Hexagonal é frequentemente considerada um padrão fundamental ou uma estratégia de implementação específica que se alinha bem com os princípios da Clean Architecture. A Clean Architecture fornece uma estrutura interna mais detalhada para o core (entidades e casos de uso) e impõe uma regra de dependência rigorosa em suas camadas concêntricas. A Arquitetura Hexagonal foca principalmente na fronteira entre o core e os sistemas externos usando portas e adaptadores, o que corresponde à camada de "Interface Adapters" na Clean Architecture.

Em essência, a Clean Architecture é um superconjunto que incorpora os benefícios de padrões como a Arquitetura Hexagonal para atingir seus objetivos de independência de framework, testabilidade, independência de UI e independência de banco de dados.
