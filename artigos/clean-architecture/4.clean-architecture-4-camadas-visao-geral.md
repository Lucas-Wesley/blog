A Clean Architecture é uma filosofia de design de software que enfatiza a separação de preocupações para criar sistemas independentes de frameworks, testáveis e adaptáveis a mudanças na UI, bancos de dados e outras agências externas.

### Círculos Concêntricos e a Regra da Dependência

O conceito central da Clean Architecture é frequentemente visualizado como um conjunto de círculos concêntricos, semelhante a uma cebola. Cada círculo representa uma camada diferente do software, sendo os círculos mais internos os mais abstratos e contendo políticas de alto nível, e os círculos mais externos sendo mais concretos e contendo mecanismos de baixo nível.

O princípio fundamental que governa essas camadas é a **Regra da Dependência**: as dependências do código-fonte só podem apontar para dentro. Isso significa que o código em um círculo interno não pode saber nada sobre o código em um círculo externo. Essa regra garante que a lógica de negócio central permaneça independente de tecnologias e frameworks externos, permitindo que eles sejam substituídos com impacto mínimo no núcleo do sistema.

### As Quatro Camadas da Clean Architecture

Embora o número de camadas possa variar dependendo da complexidade da aplicação, a Clean Architecture tipicamente define quatro camadas primárias:

1.  **Entities (Camada mais interna):**
    *   Esta camada contém as regras de negócio corporativas e a lógica de negócio central.
    *   Consiste em objetos de domínio (entidades) que encapsulam os aspectos mais fundamentais e estáveis do sistema.
    *   Esta camada é a mais abstrata e é a menos provável de mudar, pois representa o problema central que o software resolve, independentemente de como ele é implementado ou interagido.
    *   Não depende de nenhuma outra camada.

2.  **Use Cases (Camada de Aplicação):**
    *   Esta camada contém as regras de negócio específicas da aplicação e orquestra o fluxo de dados para e das entidades.
    *   Define e implementa os casos de uso do sistema, que são as ações ou funcionalidades específicas que a aplicação fornece.
    *   Mudanças nesta camada não devem afetar as entidades, nem esta camada deve ser afetada por mudanças em elementos externos como a UI ou o banco de dados.

3.  **Interface Adapters (Adaptadores de Interface):**
    *   Esta camada contém adaptadores que convertem dados entre os formatos convenientes para as camadas internas (entidades e casos de uso) e os formatos convenientes para agências externas (como bancos de dados ou frameworks web).
    *   Inclui componentes como Presenters, Views e Controllers (em uma arquitetura MVC), bem como gateways e mappers de dados.
    *   Esta camada atua como um limite, isolando as camadas internas dos detalhes de sistemas externos.

4.  **Frameworks & Drivers (Camada mais externa):**
    *   Esta camada compreende frameworks externos, ferramentas e dispositivos com os quais a aplicação interage, como bancos de dados, frameworks web, frameworks de UI e outros sistemas externos.
    *   Contém o "código de cola" que conecta esses elementos externos às camadas internas.
    *   Esta camada é onde todos os "detalhes" vão, garantindo que as mudanças nas tecnologias externas tenham impacto mínimo no núcleo da aplicação.

Ao entender e aplicar essas quatro camadas, os desenvolvedores podem construir sistemas que são robustos, fáceis de testar e adaptáveis às necessidades futuras, protegendo a lógica de negócio central da volatilidade do mundo exterior.
